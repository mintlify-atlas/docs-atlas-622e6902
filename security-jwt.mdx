The `quarkus-smallrye-jwt` extension integrates [SmallRye JWT](https://github.com/smallrye/smallrye-jwt) into Quarkus to implement JSON Web Token (JWT) security in compliance with the MicroProfile JWT specification. It lets you verify incoming JWT bearer tokens, map their claims to a `SecurityIdentity`, and secure HTTP endpoints with role-based access control. The companion `quarkus-smallrye-jwt-build` extension adds a fluent API for generating signed and encrypted tokens.

<Note>
`quarkus-oidc` also supports JWT bearer token authentication and uses `smallrye-jwt` internally to represent tokens as `JsonWebToken`. The key difference is that `quarkus-oidc` requires a live OIDC provider to fetch and rotate verification keys, while `quarkus-smallrye-jwt` can verify tokens with a static local public key. Use `quarkus-smallrye-jwt` when you control the token issuer yourself or do not have a full OIDC provider. Use `quarkus-oidc` when integrating with Keycloak, Auth0, or another OIDC-compliant provider.
</Note>

## Verifying JWTs

### Configuration

Provide a public key for signature verification and configure the expected issuer:

```properties
# Location of the RSA public key on the classpath
mp.jwt.verify.publickey.location=publicKey.pem

# Expected value of the "iss" claim
mp.jwt.verify.issuer=https://example.com/issuer

# Include the PEM file in the native executable
quarkus.native.resources.includes=publicKey.pem
```

You can also inline the key directly:

```properties
mp.jwt.verify.publickey=MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...
mp.jwt.verify.issuer=https://example.com/issuer
```

If your application runs in dev or test mode and you have not configured a key, Quarkus automatically generates an RSA key pair. The public key is used for verification; the private key is available to sign tokens with `Jwt.upn("Alice").sign()`.

### Securing endpoints

Use standard Jakarta security annotations on REST endpoints and CDI beans:

```java
import jakarta.annotation.security.RolesAllowed;
import jakarta.annotation.security.PermitAll;
import jakarta.inject.Inject;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import jakarta.ws.rs.Produces;
import jakarta.ws.rs.core.MediaType;
import org.eclipse.microprofile.jwt.JsonWebToken;

@Path("/secured")
public class TokenSecuredResource {

    @Inject
    JsonWebToken jwt;

    @GET
    @Path("/permit-all")
    @PermitAll
    @Produces(MediaType.TEXT_PLAIN)
    public String publicEndpoint() {
        String name = jwt.getClaimNames() != null
            ? jwt.getName()
            : "anonymous";
        return "hello " + name;
    }

    @GET
    @Path("/roles-allowed")
    @RolesAllowed({"User", "Admin"})
    @Produces(MediaType.TEXT_PLAIN)
    public String protectedEndpoint() {
        return "hello " + jwt.getName()
            + ", birthdate: " + jwt.getClaim("birthdate");
    }

    @GET
    @Path("/admin-only")
    @RolesAllowed("Admin")
    @Produces(MediaType.TEXT_PLAIN)
    public String adminEndpoint() {
        return "admin access for " + jwt.getSubject();
    }
}
```

Roles are mapped from the `groups` claim in the JWT by default. A token with `"groups": ["User", "Admin"]` grants both roles.

## Injecting claims

### With JsonWebToken

`JsonWebToken` is injectable in `@ApplicationScoped`, `@Singleton`, and `@RequestScoped` beans:

```java
import org.eclipse.microprofile.jwt.JsonWebToken;
import jakarta.inject.Inject;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;

@Path("/me")
public class UserResource {

    @Inject
    JsonWebToken jwt;

    @GET
    public String currentUser() {
        return jwt.getName()          // "upn" or "preferred_username" or "sub"
            + " | sub: " + jwt.getSubject()
            + " | iss: " + jwt.getIssuer();
    }
}
```

### With @Claim

Individual claims can be injected with `@Claim`. The bean must be `@RequestScoped` when injecting claims as simple types:

```java
import jakarta.enterprise.context.RequestScoped;
import jakarta.inject.Inject;
import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import org.eclipse.microprofile.jwt.Claim;
import org.eclipse.microprofile.jwt.Claims;
import org.eclipse.microprofile.jwt.JsonWebToken;

@Path("/profile")
@RequestScoped
public class ProfileResource {

    @Inject
    JsonWebToken jwt;

    @Inject
    @Claim(standard = Claims.birthdate)
    String birthdate;

    @Inject
    @Claim(standard = Claims.email)
    String email;

    @GET
    public String profile() {
        return jwt.getName() + " | " + email + " | born: " + birthdate;
    }
}
```

<Note>
`@RequestScoped` is required when injecting claims directly as `String` or other simple types. If you only inject `JsonWebToken`, you can use `@ApplicationScoped` or `@Singleton` as well.
</Note>

## Generating JWTs

Use the `Jwt` builder from `quarkus-smallrye-jwt-build` to create signed tokens:

```java
import io.smallrye.jwt.build.Jwt;
import java.util.Arrays;
import java.util.HashSet;
import org.eclipse.microprofile.jwt.Claims;

public class TokenGenerator {

    public String generateToken() {
        return Jwt.issuer("https://example.com/issuer")
                .upn("jdoe@example.com")
                .groups(new HashSet<>(Arrays.asList("User", "Admin")))
                .claim(Claims.birthdate.name(), "2001-07-13")
                .sign();  // signs with the key at smallrye.jwt.sign.key.location
    }
}
```

Set the signing key location:

```properties
smallrye.jwt.sign.key.location=privateKey.pem
```

Generate a key pair with OpenSSL if you need one:

```bash
openssl genrsa -out rsaPrivateKey.pem 2048
openssl rsa -pubout -in rsaPrivateKey.pem -out publicKey.pem
openssl pkcs8 -topk8 -nocrypt -inform pem -in rsaPrivateKey.pem -outform pem -out privateKey.pem
```

### Inner-signed and encrypted tokens

For additional security, you can generate tokens with encrypted claims:

```java
// Sign then encrypt (nested JWT)
String token = Jwt.upn("alice")
        .groups("user")
        .innerSign()    // sign with smallrye.jwt.sign.key.location
        .encrypt();     // encrypt with smallrye.jwt.encrypt.key.location

// Encrypt only (no inner signature)
String encrypted = Jwt.upn("alice")
        .claim("secret", "value")
        .encrypt();
```

## Key configuration properties

### MicroProfile JWT properties

| Property | Default | Description |
|---|---|---|
| `mp.jwt.verify.publickey` | — | Inline public key text for signature verification |
| `mp.jwt.verify.publickey.location` | — | Classpath or URL path to the verification key |
| `mp.jwt.verify.publickey.algorithm` | `RS256` | Signature algorithm; set to `ES256` for elliptic curve keys |
| `mp.jwt.verify.issuer` | — | Expected `iss` claim value |
| `mp.jwt.verify.audiences` | — | Comma-separated list of allowed `aud` values |
| `mp.jwt.verify.clock.skew` | `60` | Seconds of tolerance for token expiry checks |
| `mp.jwt.token.header` | `Authorization` | Header to read the token from |
| `mp.jwt.token.cookie` | — | Cookie name when `mp.jwt.token.header=Cookie` |
| `mp.jwt.decrypt.key.location` | — | Path to the private decryption key |

### SmallRye JWT properties

| Property | Default | Description |
|---|---|---|
| `smallrye.jwt.sign.key.location` | — | Path to the private signing key |
| `smallrye.jwt.verify.key.location` | — | Alternative to `mp.jwt.verify.publickey.location`; supports secret keys in JWK format |
| `smallrye.jwt.verify.algorithm` | — | Use for symmetric algorithms (`HS256`, `HS384`, `HS512`) |
| `smallrye.jwt.path.groups` | — | Path to groups claim (e.g., `realm/groups`) |
| `smallrye.jwt.decrypt.key.location` | — | Private decryption key location (deprecated; prefer `mp.jwt.decrypt.key.location`) |
| `smallrye.jwt.blocking-authentication` | `false` | Set to `true` to prevent event loop blocking during remote key refresh |
| `quarkus.smallrye-jwt.blocking-authentication` | `false` | Quarkus-specific wrapper for the blocking authentication setting |

## Supported public key formats

Public keys for signature verification can be provided in any of these formats:

- PKCS#8 PEM (`-----BEGIN PUBLIC KEY-----`)
- JSON Web Key (JWK)
- JSON Web Key Set (JWKS)
- JWK Base64 URL encoded
- JWKS Base64 URL encoded

## Testing with @TestSecurity and @JwtSecurity

Add the test dependency:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-test-security-jwt</artifactId>
    <scope>test</scope>
</dependency>
```

Use `@TestSecurity` to set the user and roles, and `@JwtSecurity` to add custom claims:

```java
import io.quarkus.test.junit.QuarkusTest;
import io.quarkus.test.security.TestSecurity;
import io.quarkus.test.security.jwt.Claim;
import io.quarkus.test.security.jwt.JwtSecurity;
import io.restassured.RestAssured;
import org.junit.jupiter.api.Test;

@QuarkusTest
public class TokenSecuredResourceTest {

    @Test
    @TestSecurity(user = "jdoe", roles = "User")
    public void testProtectedEndpoint() {
        RestAssured.when().get("/secured/roles-allowed")
                .then().statusCode(200);
    }

    @Test
    @TestSecurity(user = "jdoe", roles = "User")
    @JwtSecurity(claims = {
            @Claim(key = "email", value = "jdoe@example.com"),
            @Claim(key = "birthdate", value = "2001-07-13")
    })
    public void testEndpointWithClaims() {
        RestAssured.when().get("/secured/roles-allowed")
                .then().statusCode(200);
    }
}
```

The `@TestSecurity` `user` property maps to `JsonWebToken.getName()`, and `roles` maps to `JsonWebToken.getGroups()`. `@JwtSecurity` is optional and adds additional claims to the test token.
