Quarkus extensions are distributed as Maven JAR artifacts. When the Quarkus build tools encounter an extension on the classpath, they identify it by the presence of specific metadata files inside the JAR. These files tell the tooling how to display the extension, which deployment artifact to load, what capabilities it provides or requires, and whether it should be conditionally activated.

This page covers each metadata file and the configuration options available in each.

<Warning>
Two metadata files share the name `quarkus-extension` but have different file extensions: `quarkus-extension.yaml` and `quarkus-extension.properties`. The YAML file is authored by the extension developer and tracked in source control. The properties file is generated by the build plugin and is generally not edited manually.
</Warning>

---

## quarkus-extension.yaml vs. quarkus-extension.properties

| File | Who writes it | When it is used |
|---|---|---|
| `META-INF/quarkus-extension.yaml` | Extension developer | Quarkus dev tools: `code.quarkus.io`, CLI, IDE wizards |
| `META-INF/quarkus-extension.properties` | Generated by the build plugin | Quarkus bootstrap and application build |

Both files live in the **runtime** artifact. The properties file is read at application build time; the YAML file is read by developer tooling when displaying extension information, generating new projects, or adding extensions to an existing project.

## META-INF/quarkus-extension.yaml

Place this template file at `runtime/src/main/resources/META-INF/quarkus-extension.yaml`. The Quarkus Maven or Gradle plugin merges it with automatically collected information (such as the artifact coordinates and Quarkus version) when the extension is built.

### Basic fields

```yaml
name: "REST"
artifact: ${project.groupId}:${project.artifactId}:${project.version}
metadata:
  short-name: "rest"
  keywords:
    - "jaxrs"
    - "web"
    - "rest"
  categories:
    - "web"
    - "reactive"
  status: "stable"
  guide: "https://quarkus.io/guides/rest"
  icon-url: "https://quarkus.io/assets/images/about/icon-reactive.svg"
  config:
    - "quarkus.rest."
```

| Field | Description |
|---|---|
| `name` | Human-readable name displayed to users in the extension catalog. |
| `artifact` | Maven coordinates of the runtime artifact. May use `${project.*}` expressions. |
| `metadata.short-name` | Short identifier used by the CLI and other tools to locate the extension quickly. |
| `metadata.keywords` | Search terms for the extension catalog. |
| `metadata.categories` | Categories the extension appears under on `code.quarkus.io`. Omitting this field keeps the extension listed but uncategorized. |
| `metadata.status` | Maturity indicator: `stable`, `preview`, or `experimental`. Set by the extension maintainer. |
| `metadata.guide` | Link to the extension's documentation page. |
| `metadata.icon-url` | URL of a square image (PNG, JPEG, WebP, or SVG) used as the extension icon in dev tools. Minimum 220 px. |
| `metadata.config` | Configuration key prefixes used by this extension. |

### Codestart binding

If your extension provides a [codestart](/extension-codestart) — starter code generated when the extension is added to a new project — declare it here:

```yaml
metadata:
  codestart:
    name: "rest"
    languages:
      - "java"
      - "kotlin"
      - "scala"
    artifact: "io.quarkus:quarkus-project-core-extension-codestarts"
```

Without this binding, the codestart will not be applied automatically when a developer picks your extension.

### Extension offerings

Extensions may be associated with one or more support offerings, such as a commercial distribution or a team's support scope. This is done by adding an `<offering-id>-support` field to the metadata:

```yaml
name: "Acme Quarkus Magic"
artifact: "com.acme.quarkus:acme-quarkus-magic:1.0.0"
metadata:
  acme-cloud-support:
    - "supported"
  team-x:
    - "tech-preview"
```

The meaning of support scopes is defined by the offering owner. Dev tools may use this data to filter or label extensions by support status.

<Note>
Extension offerings were introduced in Quarkus 3.26.0 and are still evolving.
</Note>

### What the final artifact contains

The Quarkus Maven plugin augments the template with additional fields before packaging the JAR:

```yaml
name: "Quarkus REST (formerly RESTEasy Reactive)"
artifact: "io.quarkus:quarkus-rest:3.8.5"
description: "A Jakarta REST implementation utilizing build time processing and Vert.x."
metadata:
  short-name: "rest"
  keywords:
    - "jaxrs"
    - "web"
    - "rest"
  categories:
    - "web"
    - "reactive"
  status: "stable"
  guide: "https://quarkus.io/guides/rest"
  built-with-quarkus-core: "3.8.5"
  requires-quarkus-core: "[3.8,)"
  minimum-java-version: "17"
  capabilities:
    provides:
      - "io.quarkus.rest"
      - "io.quarkus.resteasy.reactive"
  extension-dependencies:
    - "io.quarkus:quarkus-rest-common"
    - "io.quarkus:quarkus-arc"
    - "io.quarkus:quarkus-vertx-http"
scm-url: "https://github.com/quarkusio/quarkus"
```

You do not need to add `built-with-quarkus-core`, `requires-quarkus-core`, `minimum-java-version`, or `extension-dependencies` manually — the build plugin generates them.

## Capabilities

Capabilities are named contracts that extensions can _provide_ or _require_. They allow the Quarkus build system to detect conflicts (two extensions providing the same capability) or missing dependencies (an extension requiring a capability that nothing provides).

Declare capabilities in `quarkus-extension.yaml` under `metadata.capabilities`:

```yaml
metadata:
  capabilities:
    provides:
      - "io.quarkus.rest"
      - "io.quarkus.resteasy.reactive"
```

To require a capability from another extension:

```yaml
metadata:
  capabilities:
    requires:
      - "io.quarkus.security"
```

If a required capability is absent when building an application, Quarkus reports a build-time error. If two extensions provide the same capability, Quarkus reports a conflict.

Capabilities are also readable from build steps, allowing a step to conditionally register behavior:

```java
@BuildStep
void registerIfCapabilityPresent(
        Capabilities capabilities,
        BuildProducer<AdditionalBeanBuildItem> beans) {
    if (capabilities.isPresent("io.quarkus.security")) {
        beans.produce(AdditionalBeanBuildItem.unremovableOf(SecuredGreetingService.class));
    }
}
```

## META-INF/quarkus-extension.properties

This file is generated by the `quarkus-extension-maven-plugin` (or the Quarkus Gradle plugin) and read by the Quarkus bootstrap mechanism at application build time. It is not intended to be edited manually.

Key properties:

| Property | Required | Description |
|---|---|---|
| `deployment-artifact` | Yes | Coordinates of the corresponding deployment artifact: `groupId:artifactId[:classifier:type]:version`. |
| `provides-capabilities` | Optional | Capabilities this extension provides (mirrors the YAML field). |
| `requires-capabilities` | Optional | Capabilities this extension requires. |
| `conditional-dependencies` | Optional | Conditional extension dependencies. |
| `dependency-condition` | Optional | Condition that must be satisfied for a conditional dependency on this extension to activate. |
| `parent-first-artifacts` | Optional | Artifacts that must be loaded by the system `ClassLoader` rather than the Quarkus class loader. |
| `excluded-artifacts` | Optional | Artifacts that must never be loaded, and must be excluded from the final package. |

### Generating the properties file

The `quarkus-extension-maven-plugin` generates this file during the `compile` phase. Configure it in the runtime module's `pom.xml`:

```xml
<plugin>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-extension-maven-plugin</artifactId>
    <version>${quarkus.version}</version>
    <executions>
        <execution>
            <phase>compile</phase>
            <goals>
                <goal>extension-descriptor</goal>
            </goals>
            <configuration>
                <deployment>
                    ${project.groupId}:${project.artifactId}-deployment:${project.version}
                </deployment>
            </configuration>
        </execution>
    </executions>
</plugin>
```

### Dev mode JVM options

An extension can pre-configure JVM options that are added to the command line when an application is launched in dev mode:

```xml
<configuration>
    <devMode>
        <jvmOptions>
            <add-modules>jdk.incubator.vector</add-modules>
            <enable-preview/>
        </jvmOptions>
        <xxJvmOptions>
            <UseThreadPriorities>false</UseThreadPriorities>
        </xxJvmOptions>
    </devMode>
</configuration>
```

When an application that depends on this extension is started with `quarkus dev`, the specified options are added to the JVM launch command. The Quarkus dev tools log a message informing the developer when an extension overrides a default option.

## Other generated metadata files

The Quarkus build plugin generates several additional metadata files in the extension JARs. These are informational:

| File | Module | Description |
|---|---|---|
| `META-INF/quarkus-config-roots.list` | Runtime and deployment | Lists Java classes annotated with `@ConfigRoot`. Generated; do not edit. |
| `META-INF/quarkus-javadoc.properties` | Runtime and deployment | Maps configuration keys to their Javadoc descriptions. Generated; do not edit. |
| `META-INF/quarkus-build-steps.list` | Deployment only | Lists classes that contain `@BuildStep` methods. Generated; do not edit. |

## Extension registry

Once published to Maven Central, extensions are discoverable through the Quarkus extension registry at [extensions.quarkus.io](https://extensions.quarkus.io). The registry is populated from Maven repositories and uses the `quarkus-extension.yaml` metadata to display extension information.

For community extensions, the [Quarkiverse Hub](https://github.com/quarkiverse) provides automated registry publication. When you scaffold an extension with `groupId=io.quarkiverse.[extensionId]`, the Quarkus tooling uses the Quarkiverse layout and the extension is automatically listed after its first release.

<Tip>
Fill in `metadata.categories` and `metadata.keywords` in your `quarkus-extension.yaml` before your first release. These fields control how your extension is organized and found on `code.quarkus.io`.
</Tip>
