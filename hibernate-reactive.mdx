---
title: "Reactive Database Access with Hibernate Reactive"
description: "Perform non-blocking database operations in Quarkus using Hibernate Reactive with Mutiny, returning Uni and Multi types instead of blocking calls."
---

Hibernate Reactive is the only reactive Jakarta Persistence implementation. It gives you the full power of an Object Relational Mapper while accessing the database through non-blocking drivers. In Quarkus, the Panache layer wraps Hibernate Reactive with the same concise API you get from Hibernate ORM with Panache — but every operation returns a `Uni<T>` or `Multi<T>` from [Mutiny](https://smallrye.io/smallrye-mutiny/) instead of a direct result.

## When to use reactive vs blocking ORM

<Warning>
Hibernate Reactive and Hibernate ORM are separate stacks and **cannot be mixed in the same application**. Choose one. If you add both `quarkus-hibernate-orm` and `quarkus-hibernate-reactive` to the same project, Quarkus will report a configuration error.
</Warning>

Use **Hibernate Reactive** when:
- Your workload is I/O-heavy with high concurrency (thousands of simultaneous requests).
- You are already building a fully reactive pipeline with Mutiny or RESTEasy Reactive.
- You want to avoid thread-per-request blocking for database calls.

Use **Hibernate ORM** (blocking) when:
- Your team is unfamiliar with reactive programming and the non-blocking mental model.
- You need features not yet supported in the reactive stack (e.g., second-level cache, certain query types).
- Concurrency requirements are modest and blocking threads is acceptable.

Using Quarkus REST (RESTEasy Reactive) does **not** require Hibernate Reactive. A reactive HTTP layer paired with blocking Hibernate ORM is a perfectly valid and common configuration.

## Setup

<Steps>
  <Step title="Add the extensions">
    Add `quarkus-hibernate-reactive-panache` and a reactive SQL driver for your database:

    <CodeGroup>

    ```xml pom.xml
    <!-- Hibernate Reactive with Panache -->
    <dependency>
        <groupId>io.quarkus</groupId>
        <artifactId>quarkus-hibernate-reactive-panache</artifactId>
    </dependency>

    <!-- Reactive driver — choose one for your database -->
    <dependency>
        <groupId>io.quarkus</groupId>
        <artifactId>quarkus-reactive-pg-client</artifactId>
    </dependency>
    ```

    ```gradle build.gradle
    // Hibernate Reactive with Panache
    implementation("io.quarkus:quarkus-hibernate-reactive-panache")

    // Reactive driver — choose one for your database
    implementation("io.quarkus:quarkus-reactive-pg-client")
    ```

    </CodeGroup>

    Available reactive driver extensions:
    - `quarkus-reactive-pg-client` — PostgreSQL
    - `quarkus-reactive-mysql-client` — MySQL and MariaDB
    - `quarkus-reactive-mssql-client` — Microsoft SQL Server
    - `quarkus-reactive-oracle-client` — Oracle
    - `quarkus-reactive-db2-client` — DB2
  </Step>
  <Step title="Configure the datasource">
    Use the `reactive.url` property instead of `jdbc.url`:

    ```properties application.properties
    quarkus.datasource.db-kind=postgresql
    quarkus.datasource.username=sarah
    quarkus.datasource.password=connor
    quarkus.datasource.reactive.url=vertx-reactive:postgresql://localhost:5432/mydatabase

    quarkus.hibernate-orm.schema-management.strategy=drop-and-create
    ```
  </Step>
  <Step title="Define a reactive entity">
    Import `PanacheEntity` from the reactive package — **not** from `quarkus-hibernate-orm-panache`:

    ```java Person.java
    import io.quarkus.hibernate.reactive.panache.PanacheEntity;
    import io.smallrye.mutiny.Uni;
    import jakarta.persistence.Entity;
    import java.time.LocalDate;
    import java.util.List;

    @Entity
    public class Person extends PanacheEntity {
        public String name;
        public LocalDate birth;
        public Status status;

        public static Uni<Person> findByName(String name) {
            return find("name", name).firstResult();
        }

        public static Uni<List<Person>> findAlive() {
            return list("status", Status.Alive);
        }

        public static Uni<Long> deleteByName(String name) {
            return delete("name", name);
        }
    }
    ```
  </Step>
</Steps>

## Reactive CRUD operations

Every operation that would return a direct value in blocking Panache now returns a `Uni`:

```java
import io.smallrye.mutiny.Uni;
import java.time.LocalDate;
import java.time.Month;
import java.util.List;

// create and persist
Person person = new Person();
person.name = "Stef";
person.birth = LocalDate.of(1910, Month.FEBRUARY, 1);
person.status = Status.Alive;

Uni<Void> persistOp = person.persist();

// once persisted, modifications are flushed automatically at transaction commit
// you do not need to call persist() again after changing fields

// delete (only when already persistent)
Uni<Void> deleteOp = person.delete();

// list all
Uni<List<Person>> allPersons = Person.listAll();

// find by primary key
Uni<Person> byId = Person.findById(23L);

// find with a shorthand filter
Uni<List<Person>> alive = Person.list("status", Status.Alive);

// count
Uni<Long> total  = Person.count();
Uni<Long> living = Person.count("status", Status.Alive);

// bulk delete
Uni<Long> deletedAlive = Person.delete("status", Status.Alive);
Uni<Long> deletedAll   = Person.deleteAll();
Uni<Boolean> deletedById = Person.deleteById(23L);

// bulk update
Uni<Integer> updated = Person.update(
    "name = 'Mortal' where status = ?1", Status.Alive
);
```

## Writing operations inside a transaction

Mutations must run inside a reactive transaction. Use `Panache.withTransaction()` rather than `@Transactional` (which is for the blocking stack):

```java PersonResource.java
import io.quarkus.hibernate.reactive.panache.Panache;
import io.smallrye.mutiny.Uni;
import jakarta.ws.rs.*;
import jakarta.ws.rs.core.*;
import java.net.URI;
import java.util.List;

@Path("/persons")
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
public class PersonResource {

    @GET
    public Uni<List<Person>> list() {
        return Person.listAll();
    }

    @GET
    @Path("/{id}")
    public Uni<Person> get(@PathParam("id") Long id) {
        return Person.findById(id);
    }

    @POST
    public Uni<Response> create(Person person) {
        return Panache.withTransaction(person::persist)
            .replaceWith(
                Response.created(URI.create("/persons/" + person.id)).build()
            );
    }

    @PUT
    @Path("/{id}")
    public Uni<Person> update(@PathParam("id") Long id, Person incoming) {
        return Panache.withTransaction(() ->
            Person.<Person>findById(id)
                .onItem().ifNull().failWith(NotFoundException::new)
                .invoke(entity -> entity.name = incoming.name)
        );
    }

    @DELETE
    @Path("/{id}")
    public Uni<Response> delete(@PathParam("id") Long id) {
        return Panache.withTransaction(() ->
            Person.deleteById(id)
        ).map(deleted -> deleted
            ? Response.noContent().build()
            : Response.status(Response.Status.NOT_FOUND).build()
        );
    }
}
```

## Pagination

Reactive `PanacheQuery` works the same way as the blocking version, but each terminal call returns a `Uni`:

```java
import io.quarkus.hibernate.reactive.panache.PanacheQuery;
import io.quarkus.panache.common.Page;
import io.smallrye.mutiny.Uni;

PanacheQuery<Person> query = Person.find("status", Status.Alive);

// set page size and get the first page
Uni<List<Person>> firstPage = query.page(Page.ofSize(25)).list();

// advance
Uni<List<Person>> secondPage = query.nextPage().list();

// total pages and total count
Uni<Integer> pageCount   = query.pageCount();
Uni<Long>    entityCount = query.count();
```

## Repository pattern

If you prefer not to put queries on the entity, implement `PanacheRepository<T>`:

```java PersonRepository.java
import io.quarkus.hibernate.reactive.panache.PanacheRepository;
import io.smallrye.mutiny.Uni;
import jakarta.enterprise.context.ApplicationScoped;
import java.util.List;

@ApplicationScoped
public class PersonRepository implements PanacheRepository<Person> {

    public Uni<Person> findByName(String name) {
        return find("name", name).firstResult();
    }

    public Uni<List<Person>> findAlive() {
        return list("status", Status.Alive);
    }
}
```

Inject and use it exactly like the active record version, wrapping writes in `Panache.withTransaction()`.
